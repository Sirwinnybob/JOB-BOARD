## 2026-04-01 - Input Validation Missing in Multiple Endpoints (Bulk Fix)
**Vulnerability:** Several endpoints (`PUT /api/pdfs/reorder`, `PUT /api/pdfs/:id/metadata`, `PUT /api/pdfs/:id/status`, and `PUT /api/delivery-schedule`) were missing strict type and value validation for inputs. They either passed arrays of unvalidated objects directly into queries, allowed object injections instead of strings, or failed to check `NaN` on `parseInt`.
**Learning:** Even internal-facing or "administrative" endpoints must strictly validate payload structures (arrays, objects) and exact string/number types. Without this, attackers can bypass string manipulation methods or corrupt SQLite databases with `[object Object]` representations.
**Prevention:** Always validate elements within arrays during bulk operations (e.g., `.every()`). Explicitly check `typeof input === 'string'` for text fields, and use `!Number.isNaN(parseInt(input))` (or `typeof input === 'number'`) for numeric fields. Validate JSON blobs by ensuring they are non-null objects and not arrays before stringification.

## 2026-04-01 - Unbounded Data Storage in Delivery Schedule
**Vulnerability:** The `PUT /api/delivery-schedule` endpoint allowed storing arbitrary JSON objects of any size (up to the Express payload limit) with any keys. It also allowed arbitrary slot identifiers, leading to potential database exhaustion and denial of service.
**Learning:** Storing user-provided JSON directly into a database without schema enforcement or sanitization is dangerous. Whitelisting keys and enforcing strict length limits on values is essential for data integrity and resource protection.
**Prevention:** Always sanitize JSON inputs before storage. Create a new object containing only explicitly validated and whitelisted fields. Implement size limits for arrays and character limits for strings. Use whitelists for key-like parameters (e.g., slot identifiers).
