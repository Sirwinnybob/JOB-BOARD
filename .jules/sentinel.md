## 2026-04-01 - Input Validation Missing in Multiple Endpoints (Bulk Fix)
**Vulnerability:** Several endpoints (`PUT /api/pdfs/reorder`, `PUT /api/pdfs/:id/metadata`, `PUT /api/pdfs/:id/status`, and `PUT /api/delivery-schedule`) were missing strict type and value validation for inputs. They either passed arrays of unvalidated objects directly into queries, allowed object injections instead of strings, or failed to check `NaN` on `parseInt`.
**Learning:** Even internal-facing or "administrative" endpoints must strictly validate payload structures (arrays, objects) and exact string/number types. Without this, attackers can bypass string manipulation methods or corrupt SQLite databases with `[object Object]` representations.
**Prevention:** Always validate elements within arrays during bulk operations (e.g., `.every()`). Explicitly check `typeof input === 'string'` for text fields, and use `!Number.isNaN(parseInt(input))` (or `typeof input === 'number'`) for numeric fields. Validate JSON blobs by ensuring they are non-null objects and not arrays before stringification.

## 2026-04-01 - Unbounded Data Storage in Delivery Schedule
**Vulnerability:** The `PUT /api/delivery-schedule` endpoint allowed storing arbitrary JSON objects of any size (up to the Express payload limit) with any keys. It also allowed arbitrary slot identifiers, leading to potential database exhaustion and denial of service.
**Learning:** Storing user-provided JSON directly into a database without schema enforcement or sanitization is dangerous. Whitelisting keys and enforcing strict length limits on values is essential for data integrity and resource protection.
**Prevention:** Always sanitize JSON inputs before storage. Create a new object containing only explicitly validated and whitelisted fields. Implement size limits for arrays and character limits for strings. Use whitelists for key-like parameters (e.g., slot identifiers).

## 2026-04-07 - Unhandled Exceptions in SQLite Transactions Cause Permanent DoS
**Vulnerability:** The `PUT /api/pdfs/:id/labels` endpoint initiated a SQLite transaction (`BEGIN TRANSACTION`) inside `db.serialize()` but did not validate the elements of the `labels` array before iterating over them with `stmt.run()`. An invalid payload (e.g. `[null]`) caused a synchronous `TypeError` inside the loop, bypassing the `ROLLBACK` logic and jumping straight to the route's outer `catch` block. This left the transaction open indefinitely, permanently locking the SQLite database for writes and causing a complete Denial of Service (DoS).
**Learning:** In `node-sqlite3`, if an exception is thrown synchronously inside a `db.serialize()` block after a transaction has started, the execution context jumps out of the block, bypassing normal error handling and `.finalize()` or `ROLLBACK` calls. Because SQLite uses a single connection lock for writes, an open transaction blocks all other write operations.
**Prevention:** Always strictly validate the structure and types of all array elements and object properties *before* initiating any database transactions. Never assume data passed to `stmt.run()` inside a loop is safe from throwing TypeErrors.

## 2026-04-11 - Unauthenticated Leakage of Admin Events via WebSockets
**Vulnerability:** The `broadcastUpdate` function in `backend/server.js` was ignoring the `adminOnly` flag when broadcasting events via WebSockets. It sent all events, including sensitive admin-specific notifications, to all connected WebSocket clients regardless of their authentication status or whether they were an admin.
**Learning:** Functions that accept an `adminOnly` flag must explicitly enforce that condition across all communication channels (both Push notifications and WebSockets). Failing to check authorization during a broadcast leads to unauthenticated information disclosure to viewers.
**Prevention:** Always validate authorization against the active `deviceSessions` map or session object for each individual client connection within the WebSocket broadcast loop before calling `client.send()`.

## 2026-04-19 - Information Disclosure in OCR Test Endpoint
**Vulnerability:** The GET endpoint `/api/ocr-test-image` was missing authentication middleware, allowing any unauthenticated user to fetch the OCR test image. This image is generated from user-uploaded PDFs, which could contain sensitive job or personal information.
**Learning:** While the POST (upload) and DELETE operations for a resource might be secured with authentication, it's easy to overlook securing the corresponding GET (read) operation, especially for temporary or test files. All CRUD operations on sensitive data must be consistently authenticated.
**Prevention:** When creating endpoints for testing or temporary files, always apply the same authentication and authorization checks as the primary resource. Conduct a review of all routes associated with a feature to ensure complete coverage.
